# Agent网页信息提取完整指南

## Shopify 与 Kickstarter 固定爬取清单

**必须**阅读 [mandatory-page-list.md](mandatory-page-list.md)，按站点类型执行固定爬取清单，确保每次审计的数据输入一致。

### 执行要点

1. **识别站点类型**：URL 含 `kickstarter.com/projects/` → Kickstarter；否则默认按 Shopify 处理
2. **按清单顺序 fetch**：S1、S2 为必需；S3 及以后按顺序尝试，404 或无法访问则标注 `status: unavailable`
3. **输出格式固定**：
   - `01-level0-source-urls.md`：表格列 `ID | Type | URL | Status`
   - `02-level0-extracts.md`：按 `## S1 — [类型]`、`## S2 — [类型]` 等分块；未获取的写 `(未获取)`
4. **不得自行增减页面类型**：清单外的页面可作为补充，但 S1–Sn 的结构必须保留

### URL 派生与 404 处理

- **Shopify**：从用户 URL 解析 `{domain}`，按 `mandatory-page-list.md` 构造 S3–S8 的 URL（如 `{domain}/pages/how-it-works`）
- **Kickstarter**：S1 为用户输入的项目 URL；S2 尝试 `{project_url}/posts`
- **404**：该页 `status: unavailable`，`02-level0-extracts.md` 对应区块写 `(未获取)`，不省略区块

## 常见问题

### 为什么Agent拿到的信息不完整？

**原因分析**：

1. **动态加载内容**
   - 价格、库存等信息可能通过JavaScript动态加载
   - web_fetch工具可能只能获取初始HTML
   - 解决方案：访问多个页面或使用分页参数

2. **页面结构复杂**
   - 电商网站可能使用复杂的CSS/JS渲染
   - 关键信息可能在iframe、弹窗或标签页中
   - 解决方案：尝试访问不同的URL变体

3. **反爬虫机制**
   - 部分网站有访问频率限制
   - 可能需要特定的User-Agent或Cookie
   - 解决方案：多次尝试，使用不同的访问策略

4. **内容截断**
   - web_fetch工具的limit参数可能截断内容
   - 单次抓取可能无法获取完整页面
   - 解决方案：使用分页抓取，增加limit参数

5. **登录墙或地区限制**
   - 部分信息需要登录才能查看
   - 可能根据地区显示不同内容
   - 解决方案：寻找公开的产品规格页面

## 多策略提取方法

### 策略1：多页面访问

**问题**：单一产品页可能信息不完整

**解决方案**：访问多个相关页面

```
页面组合：
1. 产品详情页 - 获取基本信息、价格、特性
2. 产品规格页 - 获取技术参数、尺寸、重量
3. 用户评价页 - 获取用户反馈、使用体验
4. 对比页面 - 获取竞品对比信息
5. 官方支持页 - 获取可靠性、保修信息
```

**执行方式**：
```
# 第一步：访问产品详情页
web_fetch(url="产品详情页URL", limit=8000)

# 第二步：访问规格页（如果有独立页面）
web_fetch(url="产品规格页URL", limit=8000)

# 第三步：访问评测页面（如果有）
web_fetch(url="评测页面URL", limit=8000)
```

### 策略2：URL变体尝试

**问题**：同一产品可能有多个URL入口

**解决方案**：尝试不同的URL格式

**电商网站URL模式**：
```
基础URL：https://www.example.com/product/123

可能的变体：
- https://www.example.com/product/123/specs
- https://www.example.com/product/123/reviews
- https://www.example.com/p/123
- https://www.example.com/item/123
- https://m.example.com/product/123（移动版）
```

**执行方式**：
```
# 尝试不同的URL变体
web_fetch(url="基础URL")
web_fetch(url="基础URL/specs")
web_fetch(url="基础URL/reviews")
```

### 策略3：分页抓取

**问题**：web_fetch的limit参数可能截断内容

**解决方案**：使用offset参数分页抓取

```
# 第一页
web_fetch(url="产品URL", offset=0, limit=4000)

# 第二页
web_fetch(url="产品URL", offset=4000, limit=4000)

# 第三页
web_fetch(url="产品URL", offset=8000, limit=4000)
```

### 策略4：搜索引擎辅助

**问题**：直接访问产品页信息不完整

**解决方案**：通过搜索引擎找到更多信息源

```
# 使用web_search搜索产品信息
web_search(query="产品型号 + specs")
web_search(query="产品型号 + review")
web_search(query="产品型号 + price")

# 访问搜索结果中的权威页面
web_fetch(url="搜索结果中的页面URL")
```

## 关键信息提取清单

### 必须提取的信息

#### 1. 基本信息
- [ ] 产品名称（完整型号）
- [ ] 品牌（用于Brand Blinding前的记录）
- [ ] 产品类别
- [ ] 发布日期/上市时间

#### 2. 价格信息
- [ ] 当前售价
- [ ] 原价/建议零售价
- [ ] 折扣信息
- [ ] 地区价格差异

**价格提取难点及解决方案**：
```
难点1：动态价格
- 解决：查找页面中的schema.org结构化数据
- 解决：查找JSON-LD或Microdata标记
- 解决：查找价格历史页面

难点2：多规格价格
- 解决：记录基础价格和不同配置的价格
- 解决：明确标注"起售价"

难点3：地区差异
- 解决：明确标注价格所属地区
- 解决：记录货币单位
```

#### 3. 技术规格
- [ ] 核心参数（尺寸、重量、材质）
- [ ] 性能参数（速度、容量、功率）
- [ ] 连接方式（接口、无线协议）
- [ ] 电池续航（如果是便携设备）
- [ ] 兼容性要求

#### 4. 用户评价
- [ ] 平均评分
- [ ] 评价数量
- [ ] 常见好评点
- [ ] 常见差评点
- [ ] 使用场景反馈

#### 5. 可靠性数据
- [ ] 保修期限
- [ ] 故障率（如果有）
- [ ] 常见问题
- [ ] 售后服务质量

#### 6. 可持续性信息
- [ ] 材质（是否环保材料）
- [ ] 能耗等级
- [ ] 可维修性
- [ ] 包装环保性

## 不同网站类型的应对策略

### 电商平台（Amazon、京东、淘宝等）

**特点**：
- 信息丰富但结构复杂
- 动态加载较多
- 可能有反爬虫机制

**策略**：
```
1. 访问产品详情页
2. 访问产品规格页（如果有独立页面）
3. 访问评价页面
4. 使用分页抓取获取完整内容
5. 查找"商品详情"或"产品参数"区域
```

**关键信息位置**：
- 价格：通常在页面顶部，可能包含税费、运费说明
- 规格：可能在"商品详情"、"参数"标签页
- 评价：独立页面或页面底部

### 官网（Apple、Dyson等）

**特点**：
- 信息准确但可能营销化
- 设计精美但可能影响抓取
- 可能有地区差异

**策略**：
```
1. 访问产品主页
2. 访问"技术规格"页面
3. 访问"对比"页面
4. 查找PDF手册或说明书
5. 查找支持页面获取可靠性信息
```

### 评测网站（The Verge、Engadget等）

**特点**：
- 信息客观深入
- 包含专业视角
- 可能有对比测试

**策略**：
```
1. 访问评测主页
2. 查找"优缺点"总结
3. 查找性能测试数据
4. 查找与竞品对比
```

## 数据完整性验证

### 验证清单

在完成信息提取后，Agent应该验证：

```markdown
## 数据完整性检查

### 基本信息（必须完整）
- [ ] 产品名称：已获取
- [ ] 价格：已获取 或 已说明缺失原因
- [ ] 主要功能：已获取

### 技术规格（至少50%）
- [ ] 核心参数：已获取___项，缺失___项
- [ ] 缺失的关键参数：[列出缺失项]

### 用户反馈（至少一项）
- [ ] 评分：已获取
- [ ] 或评价数量：已获取
- [ ] 或典型反馈：已获取

### 可靠性数据（可选但重要）
- [ ] 保修信息：已获取 或 未找到
- [ ] 故障数据：已获取 或 未找到
```

### 缺失信息处理

**当关键信息缺失时**：

1. **价格缺失**
   ```
   解决方案：
   - 尝试访问购物车页面（可能显示价格）
   - 搜索"产品名称 + price"
   - 查找其他电商平台的同一产品
   - 在报告中明确标注"价格信息缺失"
   ```

2. **技术规格缺失**
   ```
   解决方案：
   - 查找官方PDF说明书
   - 搜索"产品型号 + specs"
   - 查找技术数据库（如GSMArena对于手机）
   - 访问竞品对比页面（可能包含规格）
   ```

3. **用户评价缺失**
   ```
   解决方案：
   - 查找第三方评测网站
   - 搜索"产品型号 + review"
   - 查找论坛讨论（Reddit、贴吧等）
   - 在报告中标注"用户反馈样本较少"
   ```

### 补充信息来源

当主要来源信息不完整时，Agent应该：

```
补充信息来源优先级：
1. 官方支持页面（可靠性数据）
2. 第三方评测网站（客观评价）
3. 用户论坛和社区（真实反馈）
4. 技术数据库（详细规格）
5. 竞品对比页面（相对优势）
```

## Agent执行建议

### 最佳实践

1. **多页面访问**：不要只依赖单一页面
2. **分步提取**：先获取基础信息，再补充细节
3. **主动验证**：提取后检查完整性，主动补充缺失信息
4. **诚实记录**：明确标注哪些信息无法获取及原因

### 执行模板

```markdown
## 产品信息提取执行记录

### 第一步：访问产品主页
- URL: [产品URL]
- 获取信息: [列出获取的信息]
- 缺失信息: [列出缺失的信息]

### 第二步：访问规格页面
- URL: [规格页URL]
- 获取信息: [补充获取的信息]
- 缺失信息: [仍未获取的信息]

### 第三步：访问评测/评价页面
- URL: [评测页URL]
- 获取信息: [补充获取的信息]
- 缺失信息: [仍未获取的信息]

### 第四步：搜索补充信息
- 搜索查询: [使用的搜索词]
- 找到的信息: [补充的信息]

### 数据完整性评估
- 基本信息：完整/部分完整/不完整
- 技术规格：完整/部分完整/不完整
- 用户反馈：完整/部分完整/不完整
- 可靠性数据：完整/部分完整/不完整

### 无法获取的信息及原因
- [信息项]: [原因说明]
```

## 注意事项

1. **不要假设信息**：如果找不到价格，不要根据其他信息推测
2. **多来源验证**：重要信息应该从多个来源验证
3. **诚实记录缺失**：在报告中明确说明哪些信息无法获取
4. **保持客观**：不要因为信息缺失而影响评估的客观性
5. **利用Agent能力**：Agent可以理解上下文，比脚本更灵活

## 与Brand Blinding的配合

在完成信息提取后，需要进行Brand Blinding处理：

```
原始信息 → Brand Blinding处理 → Brand-Blinded信息

保留：技术规格、性能数据、用户评价
移除：品牌名称、营销话术、情感化描述
```

详细说明见 [references/defluff-guide.md](defluff-guide.md)
